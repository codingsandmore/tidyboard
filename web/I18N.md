# Internationalization (i18n)

Tidyboard uses [next-intl](https://next-intl-docs.vercel.app/) for internationalization, configured for the Next.js App Router.

## Supported locales

| Code | Language |
|------|----------|
| `en` | English (default) |
| `de` | Deutsch |

## How it works

### Locale detection (server-side)

`src/i18n/request.ts` runs on every server request via `getRequestConfig`. It:

1. Checks the `tb-locale` cookie (set by the in-app locale switcher).
2. Falls back to the `Accept-Language` HTTP header (browser preference).
3. Falls back to `en` if neither matches a supported locale.

There is **no URL prefix** (`/en/`, `/de/`). The locale is stored in a cookie and applied transparently.

### Provider tree

```
RootLayout (async server component)
  ‚îî‚îÄ I18nProvider (NextIntlClientProvider, client)
       ‚îî‚îÄ ThemeProvider
            ‚îî‚îÄ ApiProvider
                 ‚îî‚îÄ {children}
```

`src/i18n/provider.tsx` wraps `NextIntlClientProvider` and also exports:
- `useLocale()` ‚Äî re-exported from `next-intl`
- `<LocaleSwitcher />` ‚Äî EN/DE segmented control that sets the `tb-locale` cookie and calls `router.refresh()`

### Message files

Located in `src/i18n/messages/`:
- `en.json` ‚Äî English (source of truth)
- `de.json` ‚Äî German (must stay in key parity with en.json)

### Using translations in a component

**Server component:**
```tsx
import { getTranslations } from "next-intl/server";

export default async function MyPage() {
  const t = await getTranslations();
  return <h1>{t("app.name")}</h1>;
}
```

**Client component:**
```tsx
"use client";
import { useTranslations } from "next-intl";

export default function MyComponent() {
  const t = useTranslations();
  return <button>{t("common.save")}</button>;
}
```

## How to add a new language

1. Copy `src/i18n/messages/en.json` to `src/i18n/messages/<code>.json`.
2. Translate all values (keep all keys identical to `en.json`).
3. Add the locale code to the `locales` array in `src/i18n/config.ts`.
4. Add a label in `LOCALE_LABELS` inside `src/i18n/provider.tsx`.
5. Run `npm test` ‚Äî the key-parity test will catch any missing keys.

## Running tests

```bash
# From web/
npm test                  # runs all tests including parity check
npm run coverage          # coverage report (target ‚â• 89%)
```

The key-parity test lives at `src/i18n/messages.test.ts` and asserts that `de.json` contains exactly the same nested keys as `en.json`, with no empty string values.
